# 一、产品概述
## 1.1 产品名称
**Local LLM Auth Proxy（简称LLAP）**

## 1.2 产品定位
LLAP是一款**纯本地运行、零外部依赖、极致高性能**的LLM代理服务，核心定位为「本地AI调用控制层」。向上层应用提供**统一OpenAI兼容接口**，向下封装多模型厂商的认证、协议转换、流量调度能力，内置多源身份认证、用户级Token驻留管理、全链路观测能力，实现应用层无感接入、安全可控的LLM调用。

对上层应用的最终呈现：一个可直接访问的本地OpenAI兼容端点 `http://localhost:{port}`

## 1.3 核心特性（核心价值锚点）
| 核心特性 | 详细描述 |
| ---- | ---- |
| ✅ 纯本地运行 | 单二进制文件交付，无任何外部依赖，离线可运行，无需容器/数据库 |
| ✅ 多认证方灵活选择 | 支持Google、OIDC、OAuth2、本地用户等多种身份认证方式，可配置化切换 |
| ✅ 用户级Token驻留 | Token全程加密存储在本地代理，永不返回客户端、永不明文落盘，从根源避免泄露 |
| ✅ 极致性能 | 单请求代理延迟增量<5ms，QPS≥10k，峰值内存占用<50MB，启动时间<100ms |
| ✅ 透明无感认证 | 上层应用无需处理任何API Key、OAuth认证逻辑，零代码改造即可接入 |
| ✅ 内置Web观测 | 自带本地管理面板，支持实时指标监控、请求日志、Token管理、用量统计 |

## 1.4 核心价值
### 面向开发者/应用接入方
- 零认证开发：无需处理多厂商API Key、OAuth等复杂认证逻辑，一行配置即可接入
- 零差异适配：无需关心不同LLM厂商的API协议差异，基于统一OpenAI标准接口开发
- 零侵入改造：现有基于OpenAI接口开发的应用，仅修改base_url即可无缝切换

### 面向系统/管理方
- 极致安全：Token全程驻留本地代理，全链路加密，无泄露风险
- 可控可管：用户级请求与Token用量管控，成本可追踪、权限可隔离
- 全链路可观测：内置Web观测面板，实时监控、排障定位、用量分析一键完成

## 1.5 典型使用场景
1. AI编码工具：VSCode插件、CLI编码助手、本地IDE集成，实现多模型无感调用
2. Agent应用开发：LangChain、AutoGPT等Agent框架/平台，统一LLM调用入口
3. SaaS本地开发环境：企业SaaS产品的本地研发测试环境，避免API Key泄露、统一管控测试用量
4. 小团队共享代理：企业/团队内部共享LLM调用能力，用户级权限隔离、用量独立统计
5. 个人本地AI工具链：个人用户统一管理多厂商API Key，所有本地AI应用统一接入，避免Key分散泄露
6. 企业内网部署：企业内部私有化LLM调用网关，对接企业SSO认证，统一管控LLM访问权限

## 1.6 文档目的
本文档为LLAP产品的完整需求定义，明确产品的功能边界、交互规则、性能指标、技术要求、验收标准，作为产品研发、测试、验收、迭代的唯一官方依据。

# 二、产品目标
## 2.1 核心战略目标
打造一款轻量、高性能、安全、开箱即用的本地LLM调用网关，成为本地/私有化LLM应用开发的标准基础设施，解决LLM调用过程中的认证安全、多厂商适配、可观测性三大核心痛点。

## 2.2 功能目标（按优先级划分）
| 优先级 | 目标分类 | 核心描述 |
| ---- | ---- | ---- |
| P0 | 统一接入层 | 提供完全兼容OpenAI标准的API端点，支持核心LLM调用接口与流式传输 |
| P0 | 核心安全能力 | 实现user-based Token驻留机制，Token全程不出代理边界，全链路加密 |
| P0 | 透明认证 | 上层应用无需处理任何认证逻辑，实现无感调用 |
| P0 | 多认证方支持 | 支持X-User-ID、本地用户、Google OIDC等多种身份认证方式，可配置切换 |
| P1 | 多模型适配 | 支持OpenAI、Anthropic Claude、Google Gemini等主流LLM厂商 |
| P1 | 全链路观测 | 内置Web UI观测面板，提供指标监控、请求日志、Token管理、用量统计能力 |
| P1 | 智能路由 | 支持多厂商/多Token的轮询、故障转移、fallback能力 |
| P1 | 多用户隔离 | 支持用户级模型访问权限、用量配额、成本管控 |
| P2 | 扩展能力 | 支持语义缓存、成本感知路由、A/B测试等高级特性 |

## 2.3 性能目标（量化可验收，极致性能优先）
| 指标项 | 核心目标 | 测试条件 |
| ---- | ---- | ---- |
| 请求延迟增量 | 平均<5ms，P99<10ms | 不含上游LLM厂商响应耗时，单机1000并发下 |
| 并发承载能力 | QPS≥10000 | 单机8核16G环境，流式/非流式混合请求 |
| 内存占用 | Go实现：初始<30MB，峰值<50MB<br>Rust实现：初始<15MB，峰值<30MB | 无请求积压、1000用户规模、1000条日志缓存场景 |
| 服务启动时间 | <100ms | 标准x86/arm架构环境，无额外插件加载 |
| 流式传输性能 | 零缓冲、零延迟透传 | SSE流式响应chunk端到端延迟<1ms |

## 2.4 非目标与边界定义（明确不做的能力，避免需求蔓延）
- ❌ 不做Agent编排与流程控制能力，仅负责LLM请求的代理转发
- ❌ 不做Prompt工程、Prompt模板管理、内容优化相关能力
- ❌ 不做LLM模型训练、微调、推理部署相关能力
- ❌ 不做云端托管SaaS服务，核心定位为本地运行的单机代理服务
- ❌ 不做复杂的RBAC权限体系，MVP阶段仅做用户级隔离，不做细粒度角色权限
- ❌ 不强制依赖Docker容器，核心交付形态为单二进制文件，容器化仅作为可选方案

# 四、核心功能需求
## 4.1 API接入层（OpenAI兼容，P0）
API接入层是LLAP的入口，核心职责是提供完全兼容OpenAI官方标准的REST API，实现上层应用的零改造接入。

### 4.1.1 核心接口支持
#### 4.1.1.1 Chat Completions接口
- 接口路径：`POST /v1/chat/completions`
- 功能描述：兼容OpenAI对话补全接口，接收客户端对话请求，转发至对应LLM厂商，返回标准化响应
- 业务规则：
  1. 完全兼容OpenAI官方入参规范，支持所有官方参数（model、messages、temperature、top_p、max_tokens、stream等）
  2. 不修改客户端传入的业务参数，仅做合法性校验，非法参数直接返回标准化错误
  3. 响应体完全对齐OpenAI官方格式，确保客户端无需修改解析逻辑
- 验收标准：
  1. 现有基于OpenAI官方SDK开发的应用，修改base_url为LLAP端点后，可无修改正常运行
  2. 非流式请求的响应体结构、字段、错误码与OpenAI官方完全一致
  3. 非法入参返回的错误格式、错误码与OpenAI官方规范对齐

#### 4.1.1.2 Embeddings接口
- 接口路径：`POST /v1/embeddings`
- 功能描述：兼容OpenAI向量嵌入接口，接收客户端嵌入请求，转发至对应LLM厂商，返回标准化向量响应
- 业务规则：
  1. 完全兼容OpenAI官方入参规范，支持input、model、encoding_format等核心参数
  2. 响应体结构、向量维度、格式与OpenAI官方完全对齐
- 验收标准：
  1. 基于OpenAI Embedding接口开发的向量应用，修改base_url后可无修改正常运行
  2. 向量输出格式、维度与对应厂商官方接口完全一致，无篡改

### 4.1.2 流式响应支持（P0）
- 功能描述：支持SSE流式响应透传，实现大模型生成内容的实时推送
- 业务规则：
  1. 完全兼容`stream=true`参数，收到流式请求后，开启与上游厂商的长连接
  2. 采用passthrough透传模式，零缓冲、零阻塞，收到上游chunk后立即转发至客户端
  3. 流式结束标识`[DONE]`与OpenAI官方规范完全对齐
  4. 支持流式请求的中断处理，客户端断开连接后，立即终止与上游厂商的连接
- 验收标准：
  1. 流式响应chunk端到端延迟<1ms，无肉眼可见的卡顿
  2. 客户端断开连接后，上游请求立即终止，无无效Token消耗
  3. 流式响应的格式、字段、结束标识与OpenAI官方完全一致

### 4.1.3 透明认证兼容（P0）
- 功能描述：实现应用层无感认证，客户端无需传入有效厂商API Key
- 业务规则：
  1. 客户端`Authorization`请求头可传入任意值（包括空值、无效值），不影响请求正常执行
  2. 代理层自动完成对应厂商的认证Token注入，无需客户端处理
  3. 标准请求示例：
    ```http
    POST http://localhost:4000/v1/chat/completions
    Authorization: Bearer anything
    Content-Type: application/json

    {
      "model": "gpt-4",
      "messages": [{"role": "user", "content": "hello"}]
    }
    ```
- 验收标准：
  1. 客户端传入无效/任意Authorization头，请求可正常执行并返回正确响应
  2. 客户端无需修改任何认证相关代码，仅修改base_url即可完成接入

## 4.2 身份认证系统（P0，核心可配置能力）
身份认证系统是实现用户级隔离、权限管控的核心，**支持多认证方灵活选择与切换**，负责用户身份的识别与校验。

### 4.2.1 核心配置规则
- 支持通过配置文件切换认证方，同一时间仅启用一种认证方式，避免冲突
- 所有认证方式最终均需解析出唯一`user_id`，作为用户隔离、Token匹配、用量统计的唯一标识
- 未通过认证的请求，统一返回401 Unauthorized标准化错误

### 4.2.2 支持的认证方（按优先级）
#### 4.2.2.1 X-User-ID头认证（MVP P0，最简模式）
- 功能描述：通过自定义请求头传递用户ID，实现零配置用户身份识别
- 业务规则：
  1. 客户端请求时携带`X-User-ID: {user_id}`请求头，代理层识别为对应用户
  2. 未携带`X-User-ID`头的请求，默认归属为`default`用户
  3. user_id支持字符串格式，长度限制1-64位，仅支持字母、数字、下划线、中划线
  4. 支持配置`allowed_user_ids`白名单，仅允许列表内的用户访问
- 验收标准：
  1. 携带不同X-User-ID的请求，可正确归属到对应用户，实现Token、用量隔离
  2. 未携带X-User-ID的请求，正确归属到default用户，正常执行
  3. 配置白名单后，非白名单用户请求返回401错误

#### 4.2.2.2 本地用户认证（P0）
- 功能描述：基于本地配置的用户数据库实现身份认证，支持账号密码、固定API Key两种模式
- 业务规则：
  1. 账号密码模式：客户端通过`/auth/login`接口传入账号密码，校验通过后返回Session Token
  2. 固定API Key模式：客户端请求时携带`Authorization: Bearer {user_api_key}`，代理层匹配对应用户
  3. 用户密码采用bcrypt哈希存储，永不明文落盘
  4. 支持用户的新增、删除、启用/禁用、密码重置
- 验收标准：
  1. 正确的账号密码/API Key可正常通过认证，解析出对应用户ID
  2. 错误的认证信息返回401错误，拒绝访问
  3. 禁用用户的认证请求直接被拒绝

#### 4.2.2.3 OIDC认证（P1，支持Google等第三方认证）
- 功能描述：基于OpenID Connect协议对接第三方身份提供商，支持Google、Azure AD、Okta等主流IDP
- 业务规则：
  1. 核心支持Google OIDC认证，可配置化切换为其他OIDC兼容的IDP
  2. 认证流程：客户端跳转至IDP登录页→登录授权→回调至LLAP→校验身份→生成Session Token→返回客户端
  3. 支持配置`allowed_domains`，仅允许指定企业域名的邮箱用户登录
  4. 支持配置`scopes`，默认必选`openid、email、profile`
  5. Session有效期可配置，支持自动刷新
- 验收标准：
  1. Google账号可正常完成登录授权，正确解析出用户ID与邮箱
  2. 非允许域名的用户登录返回403错误，拒绝访问
  3. Session过期后自动失效，需重新登录

#### 4.2.2.4 OAuth2认证（P1）
- 功能描述：基于OAuth2协议对接第三方认证平台，支持GitHub、GitLab等
- 业务规则：
  1. 支持配置客户端ID、客户端密钥、授权范围、回调地址
  2. 支持配置`allowed_organizations`，仅允许指定组织内的用户登录
  3. 认证通过后生成Session Token，绑定对应用户ID
- 验收标准：
  1. 对应平台账号可正常完成授权登录，正确解析用户信息
  2. 非允许组织的用户登录返回403错误

#### 4.2.2.5 Session Token认证（P1，推荐安全模式）
- 功能描述：通过Bearer Session Token实现用户身份认证，适配所有上游认证方式
- 业务规则：
  1. 客户端请求时携带`Authorization: Bearer {session_token}`，代理层解析session获取用户身份
  2. Session结构定义：
    ```json
    {
      "session_id": "唯一会话ID",
      "user_id": "归属用户ID",
      "expires_at": "过期时间戳（秒级）",
      "created_at": "创建时间戳（秒级）"
    }
    ```
  3. 仅支持未过期的有效session，过期session请求返回401未授权错误
  4. 支持session的创建、销毁、过期自动清理
  5. 支持内存/SQLite两种存储方式
- 验收标准：
  1. 有效session请求可正确识别用户身份，正常执行请求
  2. 过期/无效session请求返回401错误，拒绝执行
  3. session过期后自动清理，无内存泄漏

## 4.3 Token管理与驻留系统（P0，核心红线特性）
Token管理系统是核心安全模块，负责多厂商Token的存储、生命周期管理、权限管控，核心实现Token全程驻留代理，永不泄露。

### 4.3.1 Token数据结构定义
```json
{
  "token_id": "唯一Token ID",
  "user_id": "归属用户ID",
  "provider": "LLM厂商标识（openai/anthropic/gemini）",
  "type": "Token类型（api_key/oauth）",
  "access_token": "加密存储的访问密钥",
  "refresh_token": "加密存储的刷新密钥（可选）",
  "expires_at": "过期时间戳（秒级，永久有效为0）",
  "created_at": "创建时间戳（秒级）",
  "updated_at": "更新时间戳（秒级）",
  "status": "状态（enabled/disabled）",
  "priority": "优先级（数字越大优先级越高）",
  "allowed_models": "允许使用的模型列表（空则全部允许）"
}
```

### 4.3.2 核心功能
#### 4.3.2.1 Token全生命周期管理（P0）
- 功能描述：支持Token的新增、删除、修改、查询、启用/禁用
- 业务规则：
  1. 支持单用户多厂商、单厂商多Token配置
  2. 仅归属用户的Token可被该用户的请求使用，用户间Token完全隔离
  3. 仅enabled状态的Token可被用于请求转发，disabled状态的Token自动跳过
  4. 支持按模型配置Token的使用权限，仅允许指定模型使用对应Token
- 验收标准：
  1. 新增Token后，对应用户的请求可正常使用该Token调用对应厂商模型
  2. 删除/禁用Token后，对应请求无法再使用该Token，返回对应错误
  3. 不同用户的Token完全隔离，无法跨用户调用

#### 4.3.2.2 Token自动刷新（P1）
- 功能描述：支持OAuth类型Token的自动刷新，避免过期失效
- 业务规则：
  1. Token过期前30分钟（可配置），自动触发刷新流程，使用refresh_token获取新的access_token
  2. 刷新成功后，自动更新Token信息，不影响业务请求
  3. 刷新失败后，记录错误日志，禁用该Token，避免无效请求
  4. 支持配置刷新阈值、最大重试次数
- 验收标准：
  1. Token过期前可自动完成刷新，业务请求无感知、无中断
  2. 刷新失败后，正确记录日志，禁用Token，返回明确错误

#### 4.3.2.3 Token驻留安全机制（P0，红线要求）
- 功能描述：确保Token全程不离开代理边界，从根源避免泄露风险
- 强制约束规则：
  1. **永不返回客户端**：任何场景下，Token的明文/密文均不会出现在返回给客户端的响应体、响应头中
  2. **永不明文落盘**：Token持久化存储必须经过AES-256-GCM加密，禁止任何形式的明文落盘，包括日志、配置文件、临时文件
  3. **仅代理内部使用**：Token仅在代理内部的请求转发环节，注入到上游厂商的请求头中，不对外暴露
  4. **全链路脱敏**：所有日志、监控、观测面板中，Token仅展示前4位+后4位，中间内容脱敏隐藏，禁止完整展示
  5. **内存安全**：Token仅在请求注入时临时解密，使用后立即清除内存中的明文数据，无残留
- 标准请求转发流程：
  ```plaintext
  client → proxy（解析用户身份、匹配对应Token）
  proxy → 临时解密Token → 注入到上游请求 → 清除内存明文
  provider → proxy → 透传响应给client
  ```
- 验收标准：
  1. 抓包分析客户端与LLAP的所有通信内容，无任何Token相关信息
  2. 查看LLAP的所有日志文件，无完整Token明文，仅脱敏展示
  3. 查看SQLite持久化文件，Token为加密状态，无法通过明文读取
  4. 服务异常崩溃时，不会产生Token明文的dump文件、临时文件

#### 4.3.2.4 Token存储方案（P0）
- 功能描述：实现Token的安全存储，兼顾性能与持久化
- 业务规则：
  1. 运行时主存储：内存存储，加密存放，保证极致访问性能
  2. 持久化存储：SQLite本地文件存储，AES-256-GCM加密落盘，服务重启后自动加载
  3. 支持配置加密密钥，用户可自定义加密密钥，提升安全性
- 验收标准：
  1. 服务重启后，可自动从SQLite加载所有Token信息，无需重新配置
  2. 无加密密钥无法解密Token数据，保证数据安全
  3. 内存访问无性能瓶颈，Token读取耗时<0.1ms

## 4.4 模型路由与负载均衡系统（P1）
路由系统负责根据请求的模型、用户、可用Token，实现智能请求分发、负载均衡、故障转移。

### 4.4.1 基础模型路由（P0）
- 功能描述：根据模型名称自动匹配对应的LLM厂商，支持自定义映射规则
- 业务规则：
  1. 默认路由规则：
     - `gpt-*` 前缀模型 → OpenAI厂商
     - `claude-*` 前缀模型 → Anthropic厂商
     - `gemini-*` 前缀模型 → Google Gemini厂商
  2. 支持用户自定义别名映射，可将自定义模型名映射到指定厂商的具体模型
  3. 自定义路由规则优先级高于默认规则
  4. 无匹配路由规则的请求，返回404模型不存在错误
- 验收标准：
  1. 传入对应前缀的模型名称，可正确路由到对应厂商
  2. 自定义别名映射可正常生效，覆盖默认规则
  3. 无匹配规则的请求，返回标准化404错误

### 4.4.2 多Token负载均衡（P1）
- 功能描述：单用户单厂商配置多Token时，支持负载均衡策略
- 业务规则：
  1. 支持轮询策略：按顺序循环使用可用Token，均衡请求压力
  2. 支持权重策略：按配置的权重比例分配请求
  3. 支持优先级策略：优先使用优先级高的Token，不可用时切换低优先级Token
  4. 自动跳过disabled状态、过期、无权限的Token
- 验收标准：
  1. 轮询策略下，多Token的请求分配数量均匀，偏差<5%
  2. 权重策略下，请求分配比例与配置权重一致
  3. 禁用/过期Token自动跳过，不会被分配请求

### 4.4.3 故障转移与Fallback（P1）
- 功能描述：请求失败时，自动切换备用Token/备用厂商，提升服务可用性
- 业务规则：
  1. 同厂商多Token场景：当前Token请求失败（限流、报错），自动切换同厂商下一个可用Token重试
  2. 跨厂商Fallback：配置了Fallback规则时，主厂商所有Token均失败后，自动切换到备用厂商对应模型重试
  3. 可配置重试次数、重试超时时间、触发重试的错误码
  4. 重试次数超过配置上限后，返回最终错误，无无限重试
- 验收标准：
  1. 主Token请求失败后，可自动切换备用Token重试，客户端无感知
  2. 主厂商所有Token失败后，可按配置切换到备用厂商，请求正常执行
  3. 重试次数超过上限后，返回标准化错误，无无限循环

## 4.5 厂商适配层（Provider Adapter，P0）
厂商适配层负责屏蔽不同LLM厂商的API协议差异，实现请求格式转换、响应标准化、Token注入。

### 4.5.1 支持的厂商列表
| 优先级 | 厂商名称 | 支持模型 |
| ---- | ---- | ---- |

| P1 | Anthropic Claude | Claude 3/3.5等全系列对话模型 |

### 4.5.2 核心职责
1. 请求格式转换：将OpenAI标准格式的请求，转换为对应厂商的API请求格式

6. 插件化设计：新增厂商无需修改核心代码，可快速扩展适配

### 4.5.3 验收标准
1. 对应厂商的模型，可通过LLAP的统一接口正常调用，返回正确响应
2. 流式/非流式请求均能正常转换，格式与OpenAI官方完全一致
3. 厂商返回的错误，可正确转换为OpenAI标准错误格式，客户端可正常解析
4. 新增厂商适配可通过插件化方式快速完成，无需修改核心代码

# 五、非功能需求
## 5.1 极致性能需求
1. 请求处理性能：单请求代理处理耗时（不含上游厂商响应）平均<5ms，P99<10ms，无请求阻塞
2. 并发承载能力：单机8核16G环境下，稳定支持≥10000 QPS的请求量，无报错、无内存溢出
3. 流式传输性能：流式响应chunk端到端透传延迟<1ms，无缓冲、无卡顿、无丢包
4. 启动性能：服务冷启动时间<100ms，实现秒级启动、开箱即用

## 5.2 资源占用需求
1. 内存占用：Go实现初始启动内存<30MB，稳定运行峰值内存<50MB；Rust实现初始启动内存<15MB，峰值内存<30MB，无内存泄漏
2. CPU占用：空闲状态CPU占用<0.5%，1000 QPS负载下CPU占用<10%
3. 磁盘占用：单二进制文件大小<50MB，无额外依赖，持久化数据默认占用<100MB
4. 资源限制：支持配置CPU、内存的使用上限，避免资源抢占影响其他应用

## 5.3 可用性与可靠性需求
1. 服务可用性：单机运行可用性≥99.95%，年 downtime<4.38小时
2. 故障自愈能力：支持请求自动重试、故障转移、Fallback，单Token/单厂商故障不影响整体服务
3. 异常处理能力：上游厂商超时、报错、限流时，快速返回标准化错误，无请求 hang 住、无内存泄漏
4. 数据一致性：Token配置、用量统计数据持久化可靠，服务异常崩溃后无数据丢失、无数据损坏

## 5.4 安全需求
1. Token安全：Token全程AES-256-GCM加密存储，仅请求注入时临时解密，使用后立即清除；永不明文落盘、永不返回客户端、永不明文日志输出
2. 访问安全：默认仅允许本机访问，支持配置访问密码、IP白名单、TLS HTTPS加密传输
3. 速率限制：支持用户级、IP级的请求速率限制，避免恶意请求、超额消耗
4. 审计日志：所有Token操作、管理员操作、异常请求均记录审计日志，不可篡改
5. 供应链安全：单二进制文件，无第三方依赖，无远程代码执行风险
6. 渗透测试要求：发布前通过安全渗透测试，无高危、中危安全漏洞

## 5.5 兼容性与可扩展性需求
1. 跨平台兼容性：支持Windows、MacOS、Linux三大主流操作系统，支持x86_64、arm64架构
2. 接口兼容性：完全兼容OpenAI v1版本API规范，兼容所有基于OpenAI SDK开发的应用
3. 插件化扩展：厂商适配层、路由策略、认证方式均采用插件化设计，支持二次开发扩展
4. 生态兼容：兼容LangChain、LlamaIndex、AutoGPT等主流AI开发框架，兼容VS Code、Cursor等主流AI编码工具

2. 零侵入接入：现有应用仅需修改base_url即可接入，无需修改任何业务代码

# 六、技术架构设计
## 6.1 总体架构
LLAP采用**数据面与控制面完全分离**的架构设计，保证数据面的极致性能，同时控制面提供丰富的管理与观测能力，整体架构分层如下：

```plaintext

│                      客户端应用 (Client)                      │
└───────────────────────────────┬─────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────────┐
│  API接入层  - OpenAI兼容接口、参数校验、协议解析、限流跨域     │
└───────────────────────────────┬─────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────────┐
│  认证解析层  - 多认证方适配、用户身份识别、Session校验        │
└───────────────────────────────┬─────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────────┐
│  Token管理层  - 加密存储、生命周期管理、用户级隔离、驻留安全  │
└───────────────────────────────┬─────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────────┐
│  路由引擎  - 模型匹配、负载均衡、故障转移、Fallback重试       │
└───────────────────────────────┬─────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────────┐
│  厂商适配层  - 协议转换、Token注入、响应标准化、错误码转换     │
└───────────────────────────────┬─────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────────┐
│                上游LLM厂商服务 (LLM Provider)                 │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  控制面  - Web UI、配置管理、指标监控、日志存储、用量统计     │
└─────────────────────────────────────────────────────────────┘
```

## 6.2 核心模块技术选型

## 6.3 数据面与控制面分离设计
### 6.3.1 数据面
- 核心定位：请求转发的核心链路，极致性能、零阻塞、零额外开销
- 核心特性：全异步非阻塞IO设计、无锁设计、内存零拷贝、与控制面完全解耦，控制面故障不影响数据面请求转发
- 性能要求：单请求处理链路耗时<2ms，无任何阻塞操作

### 6.3.2 控制面
- 核心定位：管理、观测、配置能力，不影响数据面核心性能
- 核心特性：独立服务链路、无侵入指标采集、异步日志写入，所有耗时操作均在独立线程执行，不占用数据面资源
- 要求：控制面重启、故障不影响数据面的请求转发

# 七、部署方案（非Docker为主，极致轻量）
LLAP核心交付形态为**单二进制可执行文件**，无任何外部依赖，无需Docker容器，支持三大主流平台原生部署，真正实现开箱即用。

## 7.1 交付形态
| 交付形态 | 适用场景 | 交付内容 |
| ---- | ---- | ---- |
| 单二进制文件 | 绝大多数本地运行场景 | 对应操作系统与架构的可执行文件（llap.exe/llap） |
| 系统安装包 | 桌面端长期运行场景 | MacOS pkg、Windows msi、Linux deb/rpm安装包 |
| Docker镜像 | 服务器容器化可选场景 | 轻量级Docker镜像，基于alpine构建，大小<50MB |

## 7.2 核心原生部署方式
### 7.2.1 Linux 部署（systemd系统服务）
- 适用场景：Linux服务器、内网部署、长期后台运行
- 部署步骤：
  1. 下载对应架构的二进制文件，放至系统目录：`sudo mv llap /usr/local/bin/llap`
  2. 赋予执行权限：`sudo chmod +x /usr/local/bin/llap`
  3. 创建systemd服务配置文件：`sudo vi /etc/systemd/system/llap.service`
  4. 配置文件内容：
    ```ini
    [Unit]
    Description=Local LLM Auth Proxy
    After=network.target

    [Service]
    Type=simple
    User=llap
    Group=llap
    ExecStart=/usr/local/bin/llap --config /etc/llap/llap.yaml
    Restart=always
    RestartSec=5s
    LimitNOFILE=1048576
    MemoryMax=100M
    CPUQuota=20%

    [Install]
    WantedBy=multi-user.target
    ```
  5. 重载systemd配置：`sudo systemctl daemon-reload`
  6. 启动服务并设置开机自启：`sudo systemctl enable --now llap`
- 优势：开机自启、资源限制、日志统一管理、崩溃自动重启、系统原生集成

### 7.2.2 MacOS 部署（launchd服务）
- 适用场景：MacOS桌面端、本地开发环境、长期后台运行
- 部署步骤：
  1. 下载对应架构的二进制文件，放至系统目录：`sudo mv llap /usr/local/bin/llap`
  2. 赋予执行权限：`sudo chmod +x /usr/local/bin/llap`
  3. 创建launchd配置文件：`vi ~/Library/LaunchAgents/com.llap.proxy.plist`
  4. 配置文件内容：
    ```xml
    <?xml version="1.0" encoding="UTF-8"?>
    <!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
    <plist version="1.0">
    <dict>
        <key>Label</key>
        <string>com.llap.proxy</string>
        <key>ProgramArguments</key>
        <array>
            <string>/usr/local/bin/llap</string>
            <string>--config</string>
            <string>~/.llap/llap.yaml</string>
        </array>
        <key>RunAtLoad</key>
        <true/>
        <key>KeepAlive</key>
        <true/>
        <key>StandardOutPath</key>
        <string>~/.llap/logs/llap.log</string>
        <key>StandardErrorPath</key>
        <string>~/.llap/logs/llap.error.log</string>
    </dict>
    </plist>
    ```
  5. 加载服务：`launchctl load ~/Library/LaunchAgents/com.llap.proxy.plist`
- 优势：系统原生集成、开机自启、低资源占用、支持用户级/系统级运行

### 7.2.3 Windows 部署（Windows服务）
- 适用场景：Windows桌面端、本地开发环境
- 部署步骤：
  1. 下载对应架构的llap.exe文件，放至`C:\Program Files\LLAP\`目录
  2. 下载nssm（Non-Sucking Service Manager）工具，用于注册Windows服务
  3. 打开管理员CMD，执行命令注册服务：`nssm install LLAP`
  4. 在弹出的窗口中，配置Path为`C:\Program Files\LLAP\llap.exe`，配置启动参数
  5. 配置服务开机自启，启动服务
- 替代方案：通过Go/Rust自带的Windows服务库实现原生服务，无需第三方工具
- 优势：后台静默运行、无控制台窗口、集成Windows事件日志、开机自启

### 7.2.4 包管理器一键安装
- 适用场景：快速安装、升级、管理
- 支持的包管理器：
  - MacOS：`brew install llap`
  - Linux（Debian/Ubuntu）：`apt install llap`
  - Linux（CentOS/RHEL）：`yum install llap`
  - Windows：`scoop install llap` / `choco install llap`
- 优势：一键安装、自动配置环境变量、后续一键升级、无需手动管理文件

## 12.2 核心API错误码规范
| 错误码 | 错误类型 | 描述 |
| ---- | ---- | ---- |
| 400 | Bad Request | 请求参数错误、格式错误 |
| 401 | Unauthorized | 身份认证失败、无效Token、Session过期 |
| 403 | Forbidden | 无模型访问权限、IP被禁止、域名不允许 |
| 404 | Not Found | 模型不存在、接口不存在 |
| 429 | Too Many Requests | 超出速率限制、配额用尽 |
| 500 | Internal Server Error | 代理服务内部错误 |
| 502 | Bad Gateway | 上游厂商服务错误 |
| 503 | Service Unavailable | 服务不可用、无可用Token |
| 504 | Gateway Timeout | 上游厂商请求超时 |
