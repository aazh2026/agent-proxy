## Context

基于 analyze-prd-gaps-v2 差距分析，需要解决三个关键差距：
1. Token 自动刷新：TokenRefresher 骨架存在但未集成
2. 管理面板认证：WebUI 骨架存在但未实现认证
3. 测试覆盖率：缺少端到端测试

这些差距影响用户体验、安全性和代码质量。本次变更将解决这些差距。

## Goals / Non-Goals

**Goals:**
- 集成 Token 自动刷新功能，实现后台 Token 刷新
- 为管理面板添加认证保护，确保安全访问
- 补充端到端测试，覆盖核心功能流程
- 提升整体代码质量和可维护性

**Non-Goals:**
- 不修改现有 API 接口或认证流程
- 不添加新的厂商适配
- 不重构现有架构
- 不进行性能优化

## Decisions

### Token 自动刷新集成
**决策**: 在 main.go 中创建并启动 TokenRefresher
**理由**: 
1. 利用现有 TokenRefresher 骨架
2. 在服务启动时初始化，确保一致性
3. 后台运行，不影响主请求流程

**替代方案考虑**:
- 按需刷新：增加请求延迟
- 独立进程：增加部署复杂度

### 管理面板认证
**决策**: 使用现有认证中间件保护 WebUI 路由
**理由**: 
1. 复用现有认证系统
2. 保持一致性
3. 最小化代码变更

**替代方案考虑**:
- 独立认证系统：增加维护成本
- 无认证：安全风险

### 测试策略
**决策**: 补充端到端集成测试
**理由**: 
1. 覆盖核心功能流程
2. 验证集成正确性
3. 提高代码质量

**替代方案考虑**:
- 纯单元测试：无法验证集成
- 手动测试：效率低

## Risks / Trade-offs

### 风险
- **Token 刷新失败**: 可能导致 Token 过期
  - **缓解措施**: 实现重试机制和错误处理
- **认证绕过**: 可能存在安全漏洞
  - **缓解措施**: 充分测试和安全审计
- **测试不充分**: 可能遗漏边界情况
  - **缓解措施**: 覆盖主要场景和边界条件

### 权衡
- **性能 vs 功能**: Token 自动刷新增加后台任务，但提升用户体验
- **安全 vs 便利**: 管理面板认证增加访问步骤，但增强安全性
- **测试 vs 开发速度**: 补充测试增加开发时间，但提高代码质量
